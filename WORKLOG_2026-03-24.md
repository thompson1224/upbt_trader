### 20. 실연동 재검증 결과

- gateway API 기준 `/api/v1/positions`가 더 이상 빈 배열이 아니고, 실제 보유 자산 `KRW-BTC`, `KRW-ETH`, `KRW-ONT` 3건을 반환하는 것을 확인했다.
- `/api/v1/portfolio/equity-curve`의 최신 스냅샷도 실보유 자산을 반영해 `openPositions: 3`, `positionValue > 0`, `equity > availableKrw` 상태로 갱신되는 것을 확인했다.
- `execution` 로그에서 업비트 `GET /v1/accounts` 성공 후 각 보유 자산에 대한 현재가 조회와 `positions.unrealized_pnl` 갱신이 반복 수행되는 것을 확인했다.
- 제한점:
  - 기존 외부 보유분에 대한 보호정책은 신호 원본이 없으므로 별도 정책 설계가 필요했다.
  - 자산곡선에는 동기화 이전의 0포지션 스냅샷이 남아 있으나, 최신 데이터부터는 실계좌 기준으로 정상 반영된다.

### 28. 업비트 429 대응 및 ticker 배치화

- 파일: `backend/libs/upbit/rest_client.py`
- 파일: `backend/apps/execution_service/main.py`
- 파일: `backend/tests/unit/test_upbit_rest_client.py`
- 업비트 REST 클라이언트에 공통 `_request_json()` 래퍼를 추가했다.
- 대응 정책:
  - `429 Too Many Requests`는 `Retry-After` 헤더 우선, 없으면 지수 백오프로 재시도
  - `5xx` / 네트워크 오류는 안전한 요청(`GET`, `DELETE` 등)만 재시도
  - 주문 생성 `POST /orders`는 중복 주문 위험 때문에 네트워크 오류 재시도를 하지 않는다
- `get_tickers(markets)`를 추가해 여러 현재가를 한 번에 조회하도록 만들었다.
- 실행 서비스는
  - SL/TP 모니터
  - 포트폴리오 스냅샷 계산
  두 경로에서 코인별 개별 ticker 호출 대신 배치 ticker 호출을 사용하도록 바꿨다.
- 목적은 호출 수 자체를 줄여 업비트 rate limit 충돌을 낮추는 것이다.

### 29. 주문/체결 재동기화 보강

  - 같은 주문 정보를 다시 읽으면 동일 `trade_uuid`를 중복 저장/반영할 위험이 있었다.

- 수정:
  - 주문 동기화 시 매 poll마다 `trades` 목록을 읽고, 이미 저장된 `trade_uuid`를 제외한 신규 체결만 반영한다.

  - 부분체결이면 `order_partially_filled`, 최종 완료면 `order_filled` 이벤트를 발행한다.
  - 체결 수량/체결대금/평균가/수수료는 `trades[].funds`, `trades[].volume` 기준으로 계산하도록 바꿨다.
- 효과:
  - 부분체결 누적 반영
  - 중복 fill 반영 방지
  - 평균체결가/실현손익 계산 안정성 개선

## 남은 운영 리스크

- Redis 기반 손익 누적은 이번 패치 이후 발생하는 체결부터 정확해진다. 과거 데이터 재구성 로직은 별도로 필요하다.
- Redis 장애 시 리스크 지표는 0으로 폴백한다. 운영에서는 별도 모니터링과 장애 대응이 필요하다.
- 업비트 키 저장은 Redis 기반으로 개선했지만, 장기적으로는 전용 secret store 또는 DB 기반 감사 추적이 더 적절하다.
- 거래소 기존 보유분은 전략 신호 없이 유입되므로, 자동 SL/TP를 다시 도입하려면 사용자 승인 플로우나 별도 포지션 provenance 저장이 먼저 필요하다.
